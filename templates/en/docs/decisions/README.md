# Decisions

Use this directory for concise ADR-style records of stable technical decisions.

## Capture a Decision When
- the team chose one architectural or engineering direction over another
- the rationale should stay searchable after the current plan is closed
- future contributors are likely to revisit the same trade-off

## Keep Each Record Short
A good decision note usually includes:
- title and date
- status
- context
- decision
- consequences

## Relationship to Other Docs
- Put temporary execution detail in `docs/plans/`.
- Put stable repository rules in `docs/standards/`.
- Put durable architecture choices here when the reasoning matters.
