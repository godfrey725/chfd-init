# CLAUDE.md

## 协作约定
- 在编辑代码前先阅读 `AGENTS.md` 和相关文档。
- 对非简单工作，在实现前先写一个简短的计划。
- 保持改动最小、局部，并与现有模式一致。
- 不要假设未记录的行为；要从代码、测试或文档中确认。
- 没有新的验证证据，不要宣称完成。

## 默认工作流
### 1. 理解
- 先阅读 `docs/index.md`。
- 按其中链接继续阅读 standards、architecture、plans、decisions 和 runbooks。
- 如果任务只涉及一个很小的区域，保持探索范围收敛。

### 2. 计划
当任务是非简单的、跨多个文件、改变行为，或存在架构风险时，写一个轻量计划。

一个好的轻量计划包括：
- goal
- constraints
- files involved
- implementation approach
- validation steps
- done condition

当工作需要多步骤跟踪、跨会话交接安全，或超过一个会话才能完成时，在 `docs/plans/active/` 下创建或更新一个可持续计划。

### 3. 实现
- 做出最小但值得做的改动。
- 遵循已记录的分层和归属边界。
- 清晰优先于巧妙。
- 除非任务明确需要，不要加入猜测性的抽象、兼容垫片或额外可配置项。

### 4. 验证
在报告成功前运行相关验证。

常见检查：
- 针对变更行为的聚焦测试
- 针对受影响区域的更广回归检查
- 在相关时运行 lint、typecheck 或 build 命令
- 对面向用户的流程或运维流程，依据 runbook 做手工检查

### 5. 文档
当行为、规则、架构或操作流程变化时更新文档。

## 命令使用
将 `.claude/commands/` 下的仓库命令文档视为默认操作约定：
- `plan.md` 用于计划要求
- `implement.md` 用于实现要求
- `verify.md` 用于验证要求
- `cleanup.md` 用于变更后小范围的可维护性清理

## 质量门槛
在结束前，确认：
- 请求的任务确实已解决
- 改动最小且容易理解
- 已重新运行验证
- 需要更新的文档和计划已更新
- 剩余风险或后续事项已明确说明

## 响应风格
在汇报工作时：
- 列出变更的文件
- 总结已验证内容
- 说明哪些内容未能验证
- 标出剩余风险或假设
- 明确区分已完成工作与建议的后续事项
