# Architecture

此目录说明系统如何组织，以便变更能够尊重既有边界。

## 何时阅读这些文档
- 一个任务跨越模块或服务边界
- 你要添加新的子系统或重大依赖
- 你不确定代码应放在哪里

## 文件
- `system-overview.md` —— 系统整体形态和主要流程
- `layering-rules.md` —— 依赖方向和边界规则
- `domain-map.md` —— 领域概念、归属以及相邻职责

## 目的
用这些文档回答：代码该放在哪里、哪些部分可以一起改、哪些边界应该保持稳定。

## 指引
- 保持架构文档简短、明确，并在边界变化时及时更新。
- 优先使用有助于代理判断代码归属的图示或要点。
- 当需要记录长期有效的架构选择时，将其写入 `../decisions/README.md` 或链接到具体决策文件。
