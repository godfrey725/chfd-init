# Layering Rules

定义仓库中允许的依赖方向。

## 核心原则
依赖应向内流向稳定的领域逻辑，而不是向外流向偶发性的实现细节。

## 规则
- UI、handler 或交付层可以依赖 application 和 domain 层。
- 应用编排层可以依赖领域逻辑和抽象接口。
- 基础设施实现可以依赖领域抽象，但不应拥有业务策略。
- 较低层的工具模块不应导入较高层的产品代码。

## 约束方式建议
- 当改动触及边界时，在评审中检查依赖方向。
- 当技术栈支持时，使用 lint 规则、导入守卫或测试覆盖来约束。
- 对有意为之的例外进行文档记录，保持其可见且可审查。

## 评审问题
- 这个改动是否引入了错误方向的依赖？
- 业务逻辑是否泄漏到了交付层或基础设施代码中？
- 是否需要在这里记录一个新的接口或边界？
